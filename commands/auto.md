# Auto Command

Run the marketing pack-local Aaron workflow (SEO/GEO + influencer + ad/ROAS + email/SEND + launch/RAMP + social/ECHO + narrative/TALE) implied by a natural-language goal, at the smallest safe depth — the natural-language front door to `/aaron-marketing:seo-geo`, `/aaron-marketing:influencer`, `/aaron-marketing:ad`, `/aaron-marketing:email`, `/aaron-marketing:launch`, `/aaron-marketing:social`, and `/aaron-marketing:narrative`. Add `--deep` for an exhaustive, phase-gated, maximum-rigor run.

## Route

- resolver across all 120 skills (16 SEO/GEO + 16 influencer + 16 paid + 16 email + 16 launch + 16 social + 16 narrative + 8 protocol): SEO/GEO through `/aaron-marketing:seo-geo` (--mode research|create|audit|track), influencer through `/aaron-marketing:influencer`, paid through `/aaron-marketing:ad`, email through `/aaron-marketing:email` (--phase setup|engage|nurture|deliver), launch through `/aaron-marketing:launch` (--phase research|assemble|mobilize|prove), organic social through `/aaron-marketing:social` (--phase explore|craft|host|observe), brand narrative & messaging through `/aaron-marketing:narrative` (--phase trace|architect|land|evaluate); or describe any goal here and it routes to the right skill

**Cross-discipline seams** — the word-sense splits the router owns (each discipline command's own **Scope edge** block holds the canonical intra-discipline hand-off detail; this list is the cross-discipline index):

- **"Launch" / "campaign" sense**: "launch a product **with creators**" → campaign-planner via `/aaron-marketing:influencer`; any other product launch (positioning, plan, launch day, Product Hunt / press / app-store, retro) → `/aaron-marketing:launch`; an email "launch campaign" (a broadcast) → `/aaron-marketing:email`; a paid "campaign launch" (turning ads on) → `/aaron-marketing:ad`.
- **"Post this" three-way split**: always-on calendar & net-new packages → social-calendar-builder / social-creative-builder via `/aaron-marketing:social`; "boost / repurpose this asset" → content-amplifier via `/aaron-marketing:influencer` (paid exec `/aaron-marketing:ad`); launch-day community posts (Product Hunt / Show HN) → community-launch-runner via `/aaron-marketing:launch`. Comment/DM/mention triage & UGC permission → engagement-inbox-manager; listening / share of voice / dark-social → `/aaron-marketing:social --phase observe`; creator deliverable review → content-reviewer via `/aaron-marketing:influencer`.
- **Narrative = message, not medium**: durable brand message (what to say / positioning / message house / voice / boilerplate / strategic narrative) → `/aaron-marketing:narrative`; a *single launch's* positioning canvas + PR-FAQ → positioning-mapper / message-house-builder via `/aaron-marketing:launch` (they read the canon when it exists); *per-channel* voice → voice-dossier-builder / channel-registry via `/aaron-marketing:social`; machine-facing entity facts → entity-optimizer; finished surface copy → each discipline's creative builder (content-writer / ad-creative-builder / email-creative-builder / social-creative-builder); "is it landing / are we drifting" → `/aaron-marketing:narrative --phase evaluate`.

## Rules

- Follow the routing contract in [references/aaron-product-api-contract.md](../references/aaron-product-api-contract.md); use [references/auto-routing-scenarios.md](../references/auto-routing-scenarios.md) as the scenario library (both under the plugin root — resolve against `${CLAUDE_PLUGIN_ROOT}` when installed). Natural-language auto-invocation is host-dependent unless `/aaron-marketing:auto` is explicit. For influencer goals, route to the matching influencer skill by name or `/aaron-marketing:influencer` phase; for paid-ads goals, route to the matching ROAS-phase skill by name or `/aaron-marketing:ad` phase; for email goals, route to the matching SEND-phase skill by name or `/aaron-marketing:email` phase; for product-launch goals, route to the matching RAMP-phase skill by name or `/aaron-marketing:launch` phase; for organic-social goals, route to the matching ECHO-phase skill by name or `/aaron-marketing:social` phase; for brand-narrative & messaging goals, route to the matching TALE-phase skill by name or `/aaron-marketing:narrative` phase (`/aaron-marketing:seo-geo` covers SEO/GEO only). Only clearly non-marketing work stops with a pack-boundary note — influencer, paid, email, launch, social, and narrative goals are in scope and are routed, never declined.
- **Memory-management exemption**: memory-management lifecycle operations (query, archive, restore, purge) operate on user-owned project memory regardless of topical scope and are NOT subject to the pack-boundary check; safety is enforced by per-operation user confirmation.
- Algorithm: parse the goal -> assign the closest scenario family -> attach risk gates -> choose the smallest `/aaron-marketing:seo-geo` mode (research / create / audit / track), `/aaron-marketing:influencer` phase, `/aaron-marketing:ad` phase, `/aaron-marketing:email` phase, `/aaron-marketing:launch` phase, `/aaron-marketing:social` phase, or chain that completes the outcome -> ask only for blocking inputs -> continue to the natural stopping point.
- Ambiguity rule: with an object (URL/domain/topic/draft) but no clear goal, run lightweight triage and pick the safest useful starting chain; with no actionable object and no outcome, ask one concise blocking question.
- Single best match, no menu: route to the one best-fit chain and run it at the smallest safe depth — do not dump a menu of skills for the user to pick from (the Ambiguity rule already covers the genuinely-ambiguous case). Route only to skills/commands that exist in this pack; never name or promise a capability the pack does not have — decline or ask instead.
- Default to a single seo-geo mode / influencer phase / ad phase / email phase / launch phase / social phase when it completes the goal; otherwise chain (for SEO/GEO: research -> create -> audit -> track) as needed.
- **`--deep`** (absorbs the former `/aaron-marketing:max`): run only when the user explicitly asks for maximum-depth, exhaustive, or stress-test work. First output a phase plan, success criteria, evidence inventory, risk gates, permission checkpoints, and stop condition. Phase order for SEO/GEO goals: preflight -> research -> audit (tech / quality / visibility / authority) -> optional memory -> create (brief / write / series) -> publish gate (`/aaron-marketing:seo-geo --mode create --publish`) -> track; for influencer/ad/email/launch/social/narrative goals, coordinate the `/aaron-marketing:influencer` / `/aaron-marketing:ad` / `/aaron-marketing:email` / `/aaron-marketing:launch` / `/aaron-marketing:social` / `/aaron-marketing:narrative` phases without bypassing their gates (launch-day-conductor never runs without a launch-readiness-auditor SHIP; social-calendar-builder's publish step never runs without a social-quality-auditor pre-publish SHIP; a flagship surface never ships without a narrative-quality-auditor consistency check). Do not enable `--deep` for ordinary broad work.
- Apply scenario-library risk gates: publish readiness, YMYL, schema factuality, batch scale, external side effects, memory/entity writes, reputation ethics, GEO visibility claims, insufficient data, technical indexation.
- Respect specialist boundaries: batch content stays chunked; audit/publish readiness require full veto-aware evidence; GEO visibility cannot promise citations; memory cleanup stays with memory-management; canonical entity writes stay with entity-optimizer.
- Do not route repository maintenance, version sync, release, permission, or governance requests through `/aaron-marketing:auto`; they fall outside the pack-local marketing workflow.
- Words such as apply, commit, release, publish, or fix are not automatic permission grants. Publish-package and readiness work routes through `/aaron-marketing:seo-geo --mode create --publish`; actual CMS/external publication, commits, releases, and repo edits require explicit confirmation.
- If input starts with old `/seo:*`, preserve arguments and return one copyable replacement: `audit-page -> /aaron-marketing:seo-geo --mode audit`, `audit-domain -> /aaron-marketing:seo-geo --mode audit --authority`, `check-technical -> /aaron-marketing:seo-geo --mode audit --tech`, `generate-schema -> /aaron-marketing:seo-geo --mode create --schema`, `keyword-research -> /aaron-marketing:seo-geo --mode research`, `optimize-meta -> /aaron-marketing:seo-geo --mode create --meta`, `report -> /aaron-marketing:seo-geo --mode track --report`, `setup-alert -> /aaron-marketing:seo-geo --mode track --alert`, `write-content -> /aaron-marketing:seo-geo --mode create`. Old `/aaron-seo-geo:*` names map to `/aaron-marketing:seo-geo --mode <name>` for research/create/audit/track (flags preserved); other names map 1:1 to `/aaron-marketing:*`.
- Return an execution summary by default: completed steps, evidence used, blockers or gate stops, artifacts produced, assumptions, and next safe action. Diagnostic route trace appears only when explicitly requested.
- Never write files unless explicitly requested and supported by the runtime.

## Output

Return inline artifacts by default. Files may be written only when the user explicitly asks and the runtime can write.
